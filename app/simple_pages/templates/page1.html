GIT

Git is a source code management technology used by DevOps. It is a free and open-source version control system that may be used to efficiently manage small to large projects. Git is a version control system that allows numerous developers to collaborate on non-linear development projects. Git was invented by Linus Torvalds in 2005 for the development of the Linux kernel.

Git is a version control system used for tracking changes in computer files. It is generally used for source code management in software development.

  • Git is used to tracking changes in the source code
  • The distributed version control tool is used for source code management
  • It allows multiple developers to work together
  • It supports non-linear development through its thousands of parallel branches

GIT Branch

In Git, a branch is used to store changes until they are ready. While the main branch (master) stays stable, you can work on a branch. You can integrate it with the main branch once you've finished your work.

In Git, a branch is analogous to a tree branch. A tree branch is connected to the trunk, which is the center section of the tree. While branches can grow and fall off, the trunk remains compact and is the only portion of the tree that can be said to be alive and well. Similarly, a Git branch allows you to continue creating and coding a new feature or software modification without affecting the rest of the project. Branches can also be thought of as a new line of development in a project. The master branch is Git's default or primary branch (similar to a trunk of the tree).

Git branches come in handy in a variety of situations during the course of a project's development. Branching, as previously stated, creates a new line of development that is distinct from or isolated from the primary stable master branch. There are numerous benefits to doing so. Assume you're working on a project with your team and you've completed a feature. You contact the customer to suggest that they look at the feature, but they are too busy to do so, so you provide them a link to the project.

GIT Merge

$ git merge < branch name >

For developers that use version control systems, merging is a standard practice. Merging commits changes to another location, whether they were produced for testing, bug corrections, or other reasons. Merging takes the contents of a source branch and merges them with the contents of a target branch. Only the target branch is altered in this operation. The source branch's history is unchanged.

GIT Commit

$ git commit -m "First Commit!"

A Git project timeline is made up of commits, which are the basic building blocks. Commits are snapshots or milestones in a Git project's timeline. The git commit command is used to make commits, which represent the current state of a project. Always commit Git Snapshots to the local repository. You are not obligated to communicate with the central repository until you are ready. Each developer's local repository serves as a buffer between their contributions and the central repository, just as the staging area does between the working directory and the project history. For Git users, this means a shift in their development model. Git developers have the option of making a change and directly committing it to the central repo, Git developers have the opportunity to accumulate commits in their local repo.

Tags

$ git tag < tagname >

This document will discuss the Git concept of tagging and the git tag command. Tags are ref's that point to specific points in Git history. Tagging is generally used to capture a point in history that is used for a marked version release (i.e. v1.0.1). A tag is like a branch that doesn’t change. Unlike branches, tags, after being created, have no further history of commits. For more info on branches visit the git branch page. This document will cover the different kind of tags, how to create tags, listing all tags, deleting tags, sharing tags, and more.
